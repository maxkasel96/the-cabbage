# Ticket: Investigate `properties://browser/ebpaap` failed requests

## Summary of findings

- No application code in this repository issues requests to `properties://browser/ebpaap` or any `properties://` scheme URL.
- The front-end only calls relative first-party API routes (for example `/api/players`, `/api/tournaments`, `/api/rules`, `/api/admin/*`), which indicates this alert is not sourced from intentional app network logic.
- This pattern is most consistent with browser-internal or extension-generated traffic being captured by monitoring, then counted as a failed request.
- Based on the above, this appears to be a monitoring classification/noise issue rather than a production user-facing outage in this app.

## Evidence collected

### 1) Direct string search for the failing URL pattern
- Searched repository for:
  - `properties://browser/ebpaap`
  - `properties://browser`
  - `properties://`
- Result: no matches in source.

### 2) Search for monitoring-specific integration/config in code
- Searched for `webeyez` and common telemetry SDK markers.
- Result: no Webeyez-specific configuration in repository.

### 3) Outbound request pattern audit
- Reviewed front-end `fetch` usage.
- Observed only same-origin relative API paths (no `properties://` scheme usage).

## Impact assessment

### Real users vs monitoring-only

**Conclusion:** likely monitoring-only.

Reasoning:
- `properties://` is not a normal app/backend endpoint scheme.
- There is no code path in this app constructing that URL.
- Such requests are commonly generated by browser internals/extensions and should not be used to infer app availability.

### Scope and frequency

Without raw Webeyez event payload access in this repository, exact counts cannot be computed here.

Expected affected population:
- Sessions running browsers/environments where internal `properties://` access is attempted and captured by instrumentation.
- Potentially clustered by browser family/version or extension profile.

Recommended way to measure in Webeyez:
- Group failed requests by `url_scheme`, `hostname`, and `request_type`.
- Compare `properties://*` failure volume vs total failed requests and vs first-party endpoint failures.
- Break down by browser, OS, and environment (prod/stage/synthetic).

## Webeyez threshold/config validation recommendations

Current alert appears over-broad if it fires on non-HTTP(S) requests.

Recommended config changes:
1. **Exclude non-actionable schemes**: ignore `properties://`, `chrome://`, `about:`, `moz-extension://`, `chrome-extension://`, `file://`.
2. **Scope to first-party domains**: alert only for app-owned hosts and API routes.
3. **Use ratio + volume gates**:
   - Trigger only when error-rate for first-party requests exceeds threshold (example: >2% over 5 minutes)
   - And minimum sample size is met (example: >=100 requests window)
4. **Segment synthetic/monitoring traffic** from real-user monitoring when evaluating incident severity.
5. **Add allowlist/denylist governance** for known noisy URL patterns (including `properties://browser/ebpaap`).

## Recommended mitigation

Immediate:
- Add suppression/exclusion rule for `properties://browser/ebpaap` and related internal schemes.
- Re-evaluate the open alert after exclusion to verify first-party error rate is unaffected.

Near term:
- Tighten Webeyez alert definitions to first-party API/network failures only.
- Add dashboard panel explicitly showing “ignored internal/browser scheme failures” so noise is visible but non-paging.
